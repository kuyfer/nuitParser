Message Server

The Message Server is a background service process handling communication with a message queuing infrastructure. The process is based on the JMS (Java Message Service) and therefore able to communicate with most queuing systems on the market. It has been configured with

  • Oracle Advanced Queuing
  • IBM MQ/Series
  • AcitiveMQ

The Message Server hosts message processing components called Beans that process incoming messages. Beans can transform the message and route them to another message queue, write to a file or perform other specific tasks. The beans and their connection to the queuing infrastructure are described in the file Quicksilver.xml in the customer directory.

Syntax

ll_jrun MessageServerMain [-help] [-trace] [<file>]

Parameter List

-helpThis help message
-traceTrace on: D = DB-Trace, Q = General Trace
<file>Use the given configuration file. Default is Quicksilver.xml in the customer directory.

Remote control

The Message Server has an integrated remote maintenance console. A telnet application is sufficient to connect to this console. The default port is 7000 but can be changed in the ops.properties file. The following table lists the Message Server Commands. Commands can also be abbreviated.

Command

Description

display 
display beans

Display statistics about all beans configured in the server.

shutdown

Terminates the Message Server. This method ensures that all links are properly closed and that all transactions are terminated.

It is not recommended to terminate the Message Server by other means.

trace 
trace list 
trace on <channels> 
trace off 
trace off <channels>

Tracing can be activated and de-activated while the server is running. Lufthansa Systems customer support will advise the required trace channels. System administrators should monitor disk space carefully when tracing is activated.

start <bean>

Starts the named bean. All beans are started when the server is initialized. Starting is only necessary when a bean was stopped.

stop <bean>

Stops the named bean. The bean is disconnected from the queuing system and stops processing messages. The bean looses all private data held internally.

Do not stop a bean unless you have been told to do so by Lufthansa Systems customer support.

help 
help <command>

List the commands available or describe the named command.

Examples for configuration

The Quicksilver.xml file defines the beans configuration. Here are some examples for configuration:

Connections

...
  <connections>
    <connection name="oracleaq-topic" type="topic">
      <description>TopicConnectionFactory der NetLine-DB fuer Oracle AQ</description>
      <factoryPath>netline://aqtopic</factoryPath>
      <username>__DBUSER__</username>
      <password>__DBPASSWORD__</password>
    </connection>

    <connection name="oracleaq-queue" type="queue">
      <description>QueueConnectionFactory der NetLine-DB fuer Oracle AQ</description>
      <factoryPath>netline://aqqueue</factoryPath>
      <username>__DBUSER__</username>
      <password>__DBPASSWORD__</password>
    </connection>

    <connection name="jms-queue" type="queue">
      <description>Queue for JMS</description>
      <factoryPath>ConnectionFactory</factoryPath>
    </connection>

    <connection name="jms-topic" type="topic">
      <description>Topic for JMS</description>
      <factoryPath>ConnectionFactory</factoryPath>
    </connection>
  </connections>
...

Dispatching messages

...
    <!-- Dispatcher for AC_PERFORMANCE_OUT_QUEUE -->
    <messageBean name="AcPerfDispatch">
      <description>Dispatch AcPerformance messages to other queues</description>
      <class>lisline.util.mq.DispatchBean</class>
      <property>
        <name>connections</name>
        <value>
          jms-queue:netline/ops/jms/queue/AC_PERFORMANCE_OUT_LOGGER

        </value>
      </property>
      <property>
        <name>transacted</name>
        <value>true</value>
      </property>
      <queue>
        <factory ref="jms-queue"/>
        <session transacted="true" acknowledgeMode="AUTO" />
        <queuePath>netline/ops/jms/queue/AC_PERFORMANCE_OUT_QUEUE</queuePath>
      </queue>
    </messageBean>

Troubleshooting

  • Check the NETLINE.msgsrv.__.log and the specific log files configured (like NETLINE.LegDetail.__.log files). The last ones should contain the flights sent to CREW within IOPS interface or to other subscribers.
  • Check with qstate the number of messages in the netline/ops/jms/queue/LEG_DETAIL_OUT_QUEUE.
  • If your observations indicate problems on Ops side, then restart the Message Server.
